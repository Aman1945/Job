# CORRECTIVE ACTION PROCEDURE
**NexusOMS Enterprise - ISO 9001:2015 Clause 10.2 | ISO 27001:2022 Clause 10.1**  
**Version**: 1.0  
**Effective Date**: February 12, 2026

---

## 1. PURPOSE

Define the process for identifying, analyzing, and eliminating the root causes of non-conformities to prevent recurrence and drive continual improvement.

---

## 2. SCOPE

Applies to all non-conformities arising from:
- Internal/external audits
- Customer complaints
- Process monitoring
- Security incidents
- Management reviews
- Employee reports

---

## 3. CORRECTIVE ACTION PROCESS

### STEP 1: IDENTIFICATION (Day 0)

**Trigger Sources**:
- Non-conformity identified
- Customer complaint received
- Audit finding raised
- Incident occurred
- Metric threshold breached

**Documentation**:
```
CA ID: CA-YYYY-MM-DD-XXX
Date Raised: _______________
Raised By: _______________
Source: Audit / Complaint / Incident / Monitoring
Related NC ID: _______________
```

---

### STEP 2: IMMEDIATE ACTION (Day 0-1)

**Purpose**: Contain the problem and prevent immediate harm

**Actions**:
- [ ] Isolate affected products/services
- [ ] Stop non-conforming process
- [ ] Notify affected parties
- [ ] Implement temporary fix
- [ ] Document actions taken

**Responsible**: Process Owner

---

### STEP 3: ROOT CAUSE ANALYSIS (Day 1-5)

**3.1 Data Collection**
- Gather all relevant information
- Interview involved personnel
- Review process documentation
- Analyze historical data
- Examine similar past incidents

**3.2 Analysis Methods**

**A. 5 Whys Method**:
```
Problem: _______________

Why 1: _______________
Why 2: _______________
Why 3: _______________
Why 4: _______________
Why 5 (Root Cause): _______________
```

**B. Fishbone Diagram** (Ishikawa):
```
Categories to Consider:
- Man (People): _______________
- Machine (Equipment): _______________
- Method (Process): _______________
- Material (Inputs): _______________
- Measurement (Data): _______________
- Environment: _______________

Root Cause: _______________
```

**C. Fault Tree Analysis** (for complex issues):
- Top event: The non-conformity
- Contributing factors: AND/OR gates
- Basic events: Root causes

**3.3 Root Cause Validation**
- Test hypothesis
- Verify with data
- Confirm with stakeholders

**Responsible**: Process Owner + Quality Manager

---

### STEP 4: CORRECTIVE ACTION PLAN (Day 5-7)

**4.1 Develop Actions**

**Corrective Actions** (eliminate root cause):
```
Action 1: _______________
  - Responsible: _______________
  - Deadline: _______________
  - Resources: _______________
  - Success Criteria: _______________

Action 2: _______________
  - Responsible: _______________
  - Deadline: _______________
  - Resources: _______________
  - Success Criteria: _______________
```

**Preventive Actions** (prevent in other areas):
```
Action 1: _______________
Action 2: _______________
```

**4.2 Risk Assessment**
- What could go wrong with the corrective action?
- Mitigation: _______________

**4.3 Approval**
- Reviewed By: Quality Manager
- Approved By: Department Head / Management (if resources required)

**Responsible**: Process Owner

---

### STEP 5: IMPLEMENTATION (Day 7-30)

**5.1 Execute Actions**
- Implement corrective actions as planned
- Update procedures/work instructions
- Provide training to affected personnel
- Communicate changes to stakeholders

**5.2 Progress Tracking**

| Action | Status | % Complete | Issues | Next Steps |
|--------|--------|------------|--------|------------|
|        |        |            |        |            |

**5.3 Documentation**
- Update process documentation
- Revise forms/templates
- Update training materials
- Record lessons learned

**Responsible**: Action Owners

---

### STEP 6: VERIFICATION (Day 30-60)

**6.1 Effectiveness Check**

**Verification Methods**:
- [ ] Review metrics/KPIs (has performance improved?)
- [ ] Conduct follow-up audit
- [ ] Monitor for recurrence
- [ ] Review customer feedback
- [ ] Analyze process data

**Verification Period**: Minimum 30 days

**6.2 Verification Results**:
```
Verification Date: _______________
Verification Method: _______________
Results: _______________

Effectiveness: 
[ ] Effective - Root cause eliminated, no recurrence
[ ] Partially Effective - Improvement seen, monitoring continues
[ ] Not Effective - Recurrence detected, re-analysis required
```

**Responsible**: Quality Manager

---

### STEP 7: CLOSURE (Day 60)

**7.1 Closure Criteria**
- [ ] All actions completed
- [ ] Effectiveness verified
- [ ] No recurrence in verification period
- [ ] Documentation updated
- [ ] Lessons learned captured

**7.2 Closure Documentation**:
```
CA ID: _______________
Closure Date: _______________
Final Status: Closed / Re-opened
Lessons Learned: _______________
Recommendations: _______________

Closed By: _______________
Approved By: _______________
```

**Responsible**: Quality Manager

---

|-------|------|--------|-------------|------------|---------|-------|----------|--------|--------------|
|       |      |        |             |            |         |       |          |        |              |

**Review Frequency**: Monthly

---

## 5. ESCALATION

### 5.1 Escalation Triggers
- Overdue actions (>30 days)
- Ineffective corrective actions
- Recurring non-conformities
- High-risk issues

### 5.2 Escalation Path
1. **Level 1**: Process Owner → Department Head
2. **Level 2**: Department Head → Quality Manager
3. **Level 3**: Quality Manager → Management

---

## 6. METRICS

Track and report:
- Number of CAs raised per month
- Average time to close CAs
- CA closure rate (on-time vs overdue)
- CA effectiveness rate
- Recurring CAs
- CAs by source (audit, complaint, etc.)
- CAs by department

**Target**:
- Closure rate: ≥ 90% on time
- Effectiveness rate: ≥ 95%
- Recurrence rate: < 5%

---

## 7. TRAINING

### 7.1 Required Training
- Root cause analysis techniques
- Corrective action process
- Problem-solving tools
- Documentation requirements

### 7.2 Target Audience
- Process owners
- Department heads
- Quality team
- Auditors

**Frequency**: Annually

---

## 8. CONTINUOUS IMPROVEMENT

### 8.1 Process Review
- Review CA procedure annually
- Analyze CA trends quarterly
- Identify systemic issues
- Implement process improvements

### 8.2 Best Practices
- Share successful CAs across departments
- Recognize effective problem-solving
- Build knowledge base of solutions
- Benchmark against industry standards
